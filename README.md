# Per-user fork studio

An educational Cloudflare Workers prototype that gives each visitor a personal fork of a base app. Users describe changes in natural language, a Think agent edits the fork's files, and the Worker rebuilds and serves the updated app.

## Productized version

The prototype's harness is extracted into a reusable package with a Vite
plugin front-end — point it at a directory of app files and it generates the
forkable host worker around them, with pluggable auth and bundling:

- [`packages/vite-plugin-forkable-worker`](packages/vite-plugin-forkable-worker) — harness runtime + Vite plugin
- [`examples/todo`](examples/todo) — this prototype's base app, consuming the plugin
- [`examples/react-notes`](examples/react-notes) — a React base app with real
  npm dependencies, exercising worker-bundler's esbuild path

A CI deploy workflow for the examples lives at
[`.github/deploy-examples.workflow.yml`](.github/deploy-examples.workflow.yml)
— activate it with
`git mv .github/deploy-examples.workflow.yml .github/workflows/deploy-examples.yml`
(it couldn't be pushed into `workflows/` directly: the automation token lacks
the `workflows` permission) and add the `CLOUDFLARE_API_TOKEN` +
`CLOUDFLARE_ACCOUNT_ID` repository secrets. A deploy that changes an
example's `app/` files is the upstream update forks pull.

The prototype below remains a self-contained reference (it does not use the
package).

## What it demonstrates

- Per-user repositories with the Artifacts binding
- Agentic file edits with `@cloudflare/think` and `Workspace`
- Runtime bundling with `@cloudflare/worker-bundler`
- Dynamic Worker loading and per-fork Durable Object facets
- Wrangler-generated binding types in `worker-configuration.d.ts`

## Run locally

```bash
npm install
npm run typegen
npm run dev
```

## Validate

```bash
npm run check
```

## Project layout

- `src/server.ts` contains the `UserApp` Durable Object: repository lifecycle, agent loop, rebuilds, commits, and app handoff.
- `src/router.ts` contains the top-level Worker routes for login, admin reset, API calls, and app forwarding.
- `src/config.ts` contains shared constants, model selection, and naming helpers.
- `src/types.ts` contains small shared app state types.
- `src/app-runtime.ts` contains dynamic Worker loading, facet mounting, asset serving, and HTML overlay injection.
- `src/base-app.ts` contains the seeded app files that every user fork starts from.
- `src/overlay.ts` contains the host-injected fork/customization panel.
- `worker-configuration.d.ts` is generated by Wrangler; rerun `npm run typegen` after changing `wrangler.jsonc`.
