# vite-plugin-forkable-worker

Turn a Cloudflare Worker app into a **forkable worker**: every visitor gets a
personal fork they can reshape with natural language, while the developer keeps
shipping the base app — and users pull those updates into their forks, with an
AI agent resolving merge conflicts.

This is the productized form of the [per-user fork studio](../../README.md)
prototype. The building blocks are all platform primitives: per-user
repositories via the **Artifacts** binding, agentic edits via
`@cloudflare/think` + **Workspace**, in-DO rebuilds via
`@cloudflare/worker-bundler`, and per-fork execution via **Worker Loader** +
Durable Object **facets**.

## Quick start

Your project needs two parts: the **app** (the seed every fork starts from)
and the **host** (generated for you).

```
my-app/
├── app/                  # the forkable app — what users get a copy of
│   ├── src/index.ts      # must export `class App extends DurableObject`
│   ├── AGENT.md          # optional: extra instructions for the edit agent
│   └── package.json
├── vite.config.ts
└── wrangler.jsonc
```

```ts
// vite.config.ts
import { defineConfig } from "vite";
import { cloudflare } from "@cloudflare/vite-plugin";
import { forkableWorker } from "vite-plugin-forkable-worker";

export default defineConfig({
  plugins: [
    forkableWorker({ appDir: "app", entry: "src/index.ts" }),
    cloudflare()
  ]
});
```

```jsonc
// wrangler.jsonc — the harness needs these bindings
{
  "name": "my-app",
  "main": ".forkable/entry.ts", // generated by the plugin
  "compatibility_date": "2026-06-11",
  "compatibility_flags": ["nodejs_compat"],
  "artifacts": [{ "binding": "ARTIFACTS", "namespace": "default", "remote": true }],
  "ai": { "binding": "AI" },
  "durable_objects": { "bindings": [{ "name": "USERAPP", "class_name": "UserApp" }] },
  "migrations": [{ "tag": "v1", "new_sqlite_classes": ["UserApp"] }],
  "worker_loaders": [{ "binding": "LOADER" }]
}
```

`vite dev` to run locally, `vite build && wrangler deploy` to ship. The plugin
validates the wrangler config at startup and prints the expected shape if
anything is missing. See [`examples/todo`](../../examples/todo) for a working
project.

## How it works

At build time the plugin inlines everything under `appDir` into a virtual
module (`virtual:forkable-worker/app`) and generates a tiny host entry:

```ts
import { createForkableWorker } from "vite-plugin-forkable-worker/runtime";
import manifest from "virtual:forkable-worker/app";

const forkable = createForkableWorker(manifest);
export class UserApp extends forkable.UserApp {}
export default forkable.handler;
```

At runtime, the harness:

- seeds a base Artifacts repo from your app files on first request
- gives each authenticated visitor a Durable Object that forks the base repo,
  clones it into the agent's Workspace, and serves the built app through the
  Worker Loader (the app's `App` DO is mounted as a facet, so its storage
  survives rebuilds)
- injects a floating fork/customize panel into the app's HTML
- runs a Think agent over the fork's files for natural-language edits, then
  rebuilds, commits, and pushes
- lets forks pull base updates by git merge, falling back to the agent when
  the merge conflicts

## The forkable app contract

The plugin cannot make arbitrary workers forkable — forks are rebuilt *inside
a Durable Object* on every edit, so the app must fit that runtime:

- the entry file exports a Durable Object class (default `App`) with an
  `async fetch(request)`
- persistence goes through `this.ctx.storage`
- **no npm dependencies** (see "Bundling" below) and no build step
- the host owns the outer chrome (login, fork panel); the app renders itself

## Extension points

### Auth (`AuthProvider`)

The default is the prototype's honor-system cookie login (`cookieAuth()`):
visitors pick a name; the name `admin` edits the base app. **Do not ship that
to real users** — nothing stops anyone from claiming any name, including the
admin's. Swap it out by pointing the plugin at a module:

```ts
forkableWorker({ auth: "./src/access-auth.ts" })
```

```ts
// src/access-auth.ts — e.g. Cloudflare Access
import type { AuthProvider } from "vite-plugin-forkable-worker/runtime";

const provider: AuthProvider = {
  identify(request) {
    const email = request.headers.get("Cf-Access-Authenticated-User-Email");
    if (!email) return null;
    return { user: email, isAdmin: email === "owner@example.com" };
  }
};
export default provider;
```

`identify` returns a stable `user` string (it names the fork's repo and DO)
plus an `isAdmin` bit; optional `handleRequest` owns login/logout/callback
routes, and optional `loginPanelHtml` renders the sign-in UI in the overlay.

### Bundling (`AppBundler`)

The default (`workerBundler()`) wraps `@cloudflare/worker-bundler`'s
dependency-free fast transform — that's what keeps per-fork rebuilds cheap in
a DO isolate, and why the agent is told not to add dependencies. To lift that
(e.g. esbuild-wasm plus a module cache for a curated dependency set),
implement the interface and point the plugin at it:

```ts
forkableWorker({ bundler: "./src/deps-bundler.ts" })
```

```ts
import type { AppBundler } from "vite-plugin-forkable-worker/runtime";

const bundler: AppBundler = {
  async build(files, { entry }) {
    // return { mainModule, modules, serveAsset } for the Worker Loader
  }
};
export default bundler;
```

Both can also be passed as code when writing the host entry by hand:
`createForkableWorker(manifest, { auth, bundler })`.

## Options

| option | default | |
| --- | --- | --- |
| `appDir` | `"app"` | directory of seed files |
| `entry` | `"src/index.ts"` | server entry inside `appDir` |
| `name` | package.json name | prefixes Artifacts repos (`<name>-base`, `<name>-user-*`) |
| `appClassName` | `"App"` | DO class the entry exports |
| `auth` | built-in cookie auth | module default-exporting an `AuthProvider` |
| `bundler` | worker-bundler transform | module default-exporting an `AppBundler` |
| `agentInstructionsFile` | `"AGENT.md"` | appended to the agent system prompt when present |
| `systemPrompt` | built-in | full replacement for the agent system prompt |
| `models` | glm-4.7-flash / gpt-5.4 | overlay's fast/capable model slugs |
| `bindings` | `ARTIFACTS`/`AI`/`USERAPP`/`LOADER` | rename the required bindings |
| `compatibilityDate`, `compatibilityFlags` | `2026-06-11`, `["nodejs_compat"]` | for dynamically loaded forks |
| `generatedEntry` | `".forkable/entry.ts"` | where the host entry is written |

For editor support of the virtual module, add
`"types": ["vite-plugin-forkable-worker/client"]` to the project tsconfig.

## Current limitations / roadmap

- **Seed updates on redeploy**: the base repo is seeded once; after that the
  admin panel edits it. The plugin already embeds a `seedVersion` content hash
  and the base DO stores it, so the next step is committing changed seed files
  to the base repo on deploy — making the developer's normal deploy the
  "upstream update" users pull.
- **Quotas/cleanup**: every visitor costs an Artifacts repo, a DO, and AI
  spend. There is no eviction or rate limiting yet.
- **Fork isolation**: dynamically loaded fork code gets no bindings (only its
  facet storage). Keep it that way.
